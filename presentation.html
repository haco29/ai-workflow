Step 1: Context & Planning

When I get a new feature request, I start by giving Cursor full context about what I'm building, including my Cursor rules for consistent standards. 

My typical prompt to Cursor:

"I need to build a trackable insights feature for legal case management. 

Please read the context:
- @src/pages/CaseTrackables.README.md 
- @src/components/CaseManagement/
- @docs/trackables-architecture.md
- @.cursor/rules/react-development.mdc (React/TS patterns & standards)
- @.cursor/rules/quality-check.mdc (quality requirements)
- @.cursor/rules/api-development.mdc (API interaction patterns)

Now create a detailed implementation plan with:
1. Small, deliverable steps
2. Testing strategy  
3. Component architecture
4. Best practices to follow"

This gives Cursor complete context about my codebase, architecture patterns, and development standards. The Cursor rules ensure consistent React/TypeScript patterns, accessibility compliance, and quality standards. The AI can then create a plan that fits perfectly with my existing code.

Step 2: Generate Progress Document

After getting the plan, I run my @progress-md.md command to create a structured progress tracking document. 

Command: @progress-md.md

Generated output: progress.md

# 🎯 Trackable Insights Feature - Progress Tracker

## 📋 Project Overview
**Objective**: Build trackable insights UI with status management
**Status**: 🔄 In Progress | **Phase**: 1 of 3 | **Progress**: 0%

## ⏳ Pending Steps
- [ ] **STEP-001**: Create TrackableOverview component
- [ ] **STEP-002**: Add filtering and search functionality  
- [ ] **STEP-003**: Implement status override modal
- [ ] **STEP-004**: Add comprehensive tests
- [ ] **STEP-005**: Update documentation

Each step is designed to be committed independently.

This creates a living document that tracks exactly what needs to be done, with clear success criteria for each step. I can work through it systematically and track progress.

Step 3: Implement Features

I work through each step in the progress document, implementing small deliverable chunks that can be committed independently. I also use Playwright to give AI browser context for testing UI components. 

Working on STEP-001: Create TrackableOverview component

Me: "Based on our plan, implement STEP-001. Follow the patterns in 
@src/components/ui/ and ensure accessibility compliance."

Cursor creates:
- TrackableOverview.tsx with proper TypeScript interfaces
- Follows established component patterns
- Includes accessibility attributes
- Uses existing design system components

For testing UI components:
Me: "@start-playwrite.md then navigate to the component page"

Playwright opens with browser context, allowing AI to:
- See the actual rendered component
- Test interactions and responsiveness
- Validate visual design and layout
- Debug styling issues in real-time

Result: One focused, testable component that fits the architecture.

Each step is small enough to understand, test, and commit. This prevents large, risky changes and makes code review much easier.

Step 5: Add Tests

I run @add-tests.md to generate user-centric tests that follow my established testing patterns and focus on real user interactions. 

Command: @add-tests.md

Generated tests:
describe('TrackableOverview', () => {
  it('allows user to see progress visualization', () => {
    render(<TrackableOverview items={mockItems} />)
    
    // User sees progress ring
    expect(screen.getByRole('progressbar')).toBeInTheDocument()
    expect(screen.getByText('75% Complete')).toBeInTheDocument()
  })
  
  it('allows user to filter by completion status', async () => {
    render(<TrackableOverview items={mockItems} />)
    
    // User selects filter
    const filter = screen.getByLabelText(/filter by status/i)
    await user.selectOptions(filter, 'completed')
    
    // User sees filtered results
    expect(screen.getByText('3 completed items')).toBeInTheDocument()
  })
})

Tests focus on what users actually do - not implementation details. Uses accessibility-focused queries that also validate my component's a11y compliance.
